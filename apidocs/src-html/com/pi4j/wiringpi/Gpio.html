033/**
034 * <[>WiringPi GPIO Control</[>
035 * 
036 * <p>
037 * Some of the functions in the WiringPi library are designed to mimic those in the Arduino Wiring
038 * system. There are relatively easy to use and should present no problems for anyone used to the
039 * Arduino system, or C programming in-general.
040 * </p>
041 * 
042 * <p>
043 * The main difference is that unlike the Arduino system, the main loop of the program is not
044 * provided for you and you need to write it yourself. This is often desirable in a Linux system
045 * anyway as it can give you access to command-line arguments and so on. See the examples page for
046 * some simple examples and a Makefile to use.
047 * </p>
048 * 
049 * <p>
050 * Before using the Pi4J library, you need to ensure that the Java VM in configured with access to
051 * the following system libraries:
052 * <ul>
053 * <li>pi4j</li>
054 * <li>wiringPi</li>
055 * </ul>
056 * <blockquote> This library depends on the wiringPi native system library.</br> (developed by
057 * Gordon Henderson @ <a href="http://wiringpi.com/">http://wiringpi.com/</a>)
058 * </blockquote>
059 * </p>
060 * 
061 * @see <a href="http://www.pi4j.com/">http://www.pi4j.com/</a>
062 * @see <a
063 *      href="http://wiringpi.com/reference/">http://wiringpi.com/reference/</a>
064 * @author Robert Savage (<a

511    /**
512     * <p>Priority, Interrupt and Thread Functions</p>
513     *
514     * <p>
515     * This attempts to shift your program (or thread in a multi-threaded program) to a higher
516     * priority and enables a real-time scheduling. The priority parameter should be from 0 (the
517     * Default) to 99 (the maximum). This won't make your program go any faster, but it will give it
518     * a bigger slice of time when other programs are running. The priority parameter works relative
519     * to others and so you can make one program priority 1 and another priority 2 and it will have
520     * the same effect as setting one to 10 and the other to 90 (as long as no other programs are
521     * running with elevated priorities)
522     * </p>
523     * 
524     * <p>
525     * The return value is 0 for success and -1 for error. If an error is returned, the program
526     * should then consult the errno global variable, as per the usual conventions.
527     * </p>
528     * 
529     * <p>
530     * Note: Only programs running as root can change their priority. If called from a non-root
531     * program then nothing happens.
532     * </p>
533     * 
534     * @see <a
535     *      href="http://wiringpi.com/reference/priority-interrupts-and-threads/">http://wiringpi.com/reference/priority-interrupts-and-threads/</a>
536     * @param priority  The priority parameter should be from 0 (the Default) to 99 (the maximum)
537     * @return The return value is 0 for success and -1 for error. If an error is returned, the
538     *         program should then consult the errno global variable, as per the usual conventions.
539     */
540    public static native int piHiPri(int priority);

543    /**
544     * <p>Priority, Interrupt and Thread Functions</p>
545     *
546     * <p>
547     * With a newer kernel patched with the GPIO interrupt handling code, you can now wait for an
548     * interrupt in your program. This frees up the processor to do other tasks while you're waiting
549     * for that interrupt. The GPIO can be set to interrupt on a rising, falling or both edges of
550     * the incoming signal.
551     * </p>
552     * <p> <b> int waitForInterrupt (int pin, int timeOut) </b> </p>
553     * 
554     * <p>
555     * When called, it will wait for an interrupt event to happen on that pin and your program will
556     * be stalled. The timeOut parameter is given in milliseconds, or can be -1 which means to wait
557     * forever.
558     * </p>
559     * 
560     * <p>
561     * Before you call waitForInterrupt, you must first initialize the GPIO pin and at present the
562     * only way to do this is to use the gpio program, either in a script, or using the system()
563     * call from inside your program.
564     * </p>
565     * 
566     * <p>
567     * e.g. We want to wait for a falling-edge interrupt on GPIO pin 0, so to setup the hardware, we
568     * need to run:
569     * 
570     * <pre>
571     * gpio edge 0 falling
572     * </pre>
573     * 
574     * </p>
575     *
576     * @deprecated Note: Jan 2013: The waitForInterrupt() function is deprecated – you should use the newer
577     *             and easier to use wiringPiISR() function.
578     *
579     * @see <a
580     *      href="http://wiringpi.com/reference/priority-interrupts-and-threads/">http://wiringpi.com/reference/priority-interrupts-and-threads/</a>
581     * @param pin The GPIO pin number. </br><i>(Depending on how wiringPi was initialized, this may
582     *            be the wiringPi pin number or the Broadcom GPIO pin number.)</i>
583     * @param timeout The number of milliseconds to wait before timing out. </br>A value of '-1' will
584     *            disable the timeout.
585     * @return The return value is -1 if an error occurred (and errno will be set appropriately), 0
586     *         if it timed out, or 1 on a successful interrupt event.
587     */
588    public static native int waitForInterrupt(int pin, int timeout);

591    /**
592     * <p>Priority, Interrupt and Thread Functions</p>
593     *
594     * <p>
595     * This function registers a function to received interrupts on the specified pin. The edgeType parameter is either
596     * INT_EDGE_FALLING, INT_EDGE_RISING, INT_EDGE_BOTH or INT_EDGE_SETUP. If it is INT_EDGE_SETUP then no
597     * initialisation of the pin will happen – it’s assumed that you have already setup the pin elsewhere
598     * (e.g. with the gpio program), but if you specify one of the other types, then the pin will be exported and
599     * initialised as specified. This is accomplished via a suitable call to the gpio utility program, so it need to
600     * be available
601     * </p>
602     *
603     * <p>
604     * The pin number is supplied in the current mode – native wiringPi, BCM_GPIO, physical or Sys modes.
605     * </p>
606     *
607     * <p>
608     * This function will work in any mode, and does not need root privileges to work.
609     * </p>
610     *
611     * <p>
612     * The function will be called when the interrupt triggers. When it is triggered, it’s cleared in the dispatcher
613     * before calling your function, so if a subsequent interrupt fires before you finish your handler, then it won’t
614     * be missed. (However it can only track one more interrupt, if more than one interrupt fires while one is being
615     * handled then they will be ignored)
616     * </p>
617     *
618     * <p>
619     * This function is run at a high priority (if the program is run using sudo, or as root) and executes
620     * concurrently with the main program. It has full access to all the global variables, open file handles
621     * and so on.
622     * </p>
623     *
624     * @see <a
